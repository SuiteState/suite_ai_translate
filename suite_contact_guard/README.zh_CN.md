# Contact Guard

Odoo 19 联系人模块的电话号码唯一性、字段权限管控与客户活跃度状态追踪。

## 概述

Contact Guard 在 `res.partner` 上添加三层互相独立的管控机制：

1. ORM 层级的电话号码唯一性校验，按公司隔离
2. 基于角色的可见性与编辑权限管控（电话、邮箱、销售员字段）
3. 每日定时任务计算的客户活跃度状态（活跃 / 警告 / 沉睡 / 流失），基于已确认的销售订单

第四个支持性特性处理 `company_id` 默认值，让单公司用户也能获得多公司用户手动配置才能得到的多公司安全行为。

四项功能均按公司配置。所有功能均不引入数据库层级的约束；强制执行发生在 ORM 层、字段层或视图层（视具体规则而定）。详见[权限 → 强制执行层级](#强制执行层级)。

## 功能特性

### 电话号码唯一性

电话号码会被规范化为纯数字（去除 `+`、前导 `00`、空格、连字符、括号），存入索引字段供查重使用。

当内部用户创建或编辑联系人时，若同公司内已存在相同电话号码，操作会被拒绝，错误信息会指明冲突的号码。

以下场景会跳过唯一性校验：

- `env.su`（系统自动化、迁移、商机转联系人同步）
- 通过标准导入向导进行的批量导入（`import_file` context）
- Public 用户（匿名自助注册访问者）
- Portal 用户（外部客户操作自身数据）
- 显式传入的 `suite_skip_phone_check` context flag

最小位数可按公司配置，默认 8 位。

### 联系人字段权限

电话、邮箱、销售员字段的可见性按角色管控。一旦联系人被分配了销售员，其电话和邮箱就只对该销售员和管理员可见。未分配销售员的联系人对所有内部用户保持可见。

销售员字段在创建时可以自由设置（这样用户能认领自己的联系人，CRM 商机转联系人时也能携带销售员信息），但创建后即被锁定。

电话编辑保护在 ORM 层强制执行，不仅仅是视图层，因此 RPC、脚本调用同样受其约束，不止 Web UI。

完整的角色权限矩阵详见下方的[**权限**](#权限)一节。

### 客户活跃度状态

按公司启用后，每日定时任务会根据每位客户最后一笔已确认销售订单的日期计算其活跃度状态：

| 状态 | 条件 |
|---|---|
| 活跃（Active） | 距上次订单天数 < 警告阈值 |
| 警告（Warning） | 警告阈值 <= 天数 < 沉睡阈值 |
| 沉睡（Sleeping） | 沉睡阈值 <= 天数 < 流失阈值 |
| 流失（Dormant） | 天数 >= 流失阈值 |

从未有过已确认销售订单的客户没有状态（空）。这一区分通过独立的 `suite_has_orders` 布尔字段实现，让"刚下单 0 天"和"从未下单"在视图中保持可区分，同时让天数字段保持为普通整数，列表视图中能正确排序。

阈值可按公司配置。状态显示在联系人表单（活跃度状态标签页）和列表视图中（带颜色装饰：黄色=警告、红色=沉睡、灰色=流失），并可用于筛选和分组。

定时任务以 1000 条为一批处理客户，关闭聊天通知。

### 多公司 `company_id` 默认值注入

在原生 Odoo 中，单公司用户的联系人表单上不显示 `company_id` 字段，且 Odoo 默认将其设为空（这意味着该联系人对所有公司全局可见）。

Contact Guard 在创建联系人时自动注入用户当前所属公司作为默认 `company_id`，这样单公司用户也能获得多公司用户手动配置才能达到的按公司隔离效果。

当联系人是因为新建 `res.company` 而被自动创建为伴随主联系人时，这项注入会被禁用。在那种情况下，Odoo 原生流程会分配正确的 company id，否则注入会用*当前激活公司*的 id 覆盖它，破坏新公司的下游路由。

## 配置

进入：

> **联系人 -> 配置 -> Contact Guard**

（默认仅系统管理员可见。）

**重要：** 保护功能默认开启，不需要配置记录就能工作。电话号码唯一性、字段权限、销售员锁定在模块安装的瞬间就开始生效。配置只是用来调整参数或选择性启用活跃度追踪。

| 功能 | 默认状态 | 配置项作用 |
|---|---|---|
| 电话号码唯一性 | 开启 | 调整最小位数（默认 8） |
| 字段权限 | 开启 | 无配置项；行为由角色固定决定 |
| 客户活跃度状态 | 关闭 | 启用每日定时任务并设置阈值 |

每家公司有独立的配置记录：

| 设置项 | 默认值 | 说明 |
|---|---|---|
| 电话最小长度 | 8 | 规范化后的最小位数 |
| 启用客户活跃度追踪 | 关闭 | 启用本公司的每日活跃度定时任务 |
| 警告阈值（天数） | 30 | 进入警告状态的天数 |
| 沉睡阈值（天数） | 60 | 进入沉睡状态的天数 |
| 流失阈值（天数） | 90 | 进入流失状态的天数 |

阈值必须满足 `0 < 警告 < 沉睡 < 流失`。

## 权限

### 涉及的角色

| 角色 | 群组 | 来源 |
|---|---|---|
| Contact Guard 管理员 | `suite_contact_guard.group_contact_guard_admin` | 本模块新增 |
| 销售经理 | `sales_team.group_sale_manager` | 原生；**蕴含** Contact Guard 管理员 |
| 联系人管理员 | `base.group_partner_manager` | 原生 |
| 采购经理 | `purchase.group_purchase_manager` | 原生 |
| 采购用户 | `purchase.group_purchase_user` | 原生 |
| 该联系人的销售员 | `partner.user_id == self.env.user` | 按记录的关系，非群组 |

蕴含关系是单向的：**销售经理 → Contact Guard 管理员**，反之不成立。单独授予 Contact Guard 管理员不会让用户成为销售经理。这样管理员可以将 Contact Guard 管理员权限分配给销售体系外的角色（财务经理、运营总监等），不会意外地放大他们在其他模块的权限。

### 各角色可执行的操作

下表中，*销售员*指被分配为该联系人销售员的用户。*SM/CGA* 指销售经理或 Contact Guard 管理员（在 Contact Guard 范畴内等价）。

| 操作 | 内部用户 | 该联系人的销售员 | 采购经理 | 联系人管理员 | SM / CGA |
|---|---|---|---|---|---|
| 在列表和看板中看到电话/邮箱列 | ❌ | ❌ | ❌ | ✅ | ✅ |
| 在表单上看到电话/邮箱*（联系人无销售员）* | ✅ | – | ✅ | ✅ | ✅ |
| 在表单上看到电话/邮箱*（联系人有销售员）* | ❌ ¹ | ✅ | ❌ ¹ | ✅ | ✅ |
| 首次填写电话（原本为空） | ✅ | ✅ | ✅ | ✅ | ✅ |
| 编辑已有电话*（联系人无销售员）* | ❌ | ❌ | ✅ | ✅ | ✅ |
| 编辑已有电话*（联系人有销售员）* | ❌ | ❌ | ❌ | ✅ | ✅ |
| **创建**联系人时设置销售员 | ✅ | ✅ | ✅ | ✅ | ✅ |
| 修改已有联系人的销售员 | ❌ | ❌ | ❌ | ❌ | ✅ |
| 看到活跃度状态（表单标签页 + 列表列） | ❌ | ❌ | ❌ | ✅ | ✅ |
| 打开 Contact Guard 配置菜单 | ❌ | ❌ | ❌ | ❌ | 仅系统管理员 |
| 写入 Contact Guard 配置 | ❌ | ❌ | ❌ | ❌ | 仅系统管理员 |

> ¹ "内部用户"和"采购经理"在联系人有销售员时，看不到表单上的电话/邮箱*值*，但这是通过 `invisible=` 隐藏的，并不是在服务端剥离。技术上熟练的用户（DevTools、自定义 RPC 客户端）仍可读取该值。列表和看板的隐藏*是*在服务端剥离的。详见下方[表单层视觉隐藏的限制](#表单层视觉隐藏的限制)。

### 强制执行层级

Contact Guard 在三个不同层级强制执行规则，强度不同：

1. **ORM 层** *（最强）* —— 电话编辑保护和销售员变更保护在 `res.partner.write` 中强制执行。所有代码路径都受约束：Web UI、RPC、脚本、自动化操作。仅在以下情况绕过：`env.su`、`import_file` context、`suite_skip_phone_check` context、public/portal 用户。
2. **字段层 `groups=`** *（强）* —— 活跃度状态字段（`suite_last_order_date`、`suite_days_since_order`、`suite_activity_status`、`suite_has_orders`）在模型上携带 `groups=`。无权限用户通过任何渠道都读不到 —— ORM 会从 `search_read`、`read`、导出、API 调用中将其剥离。
3. **视图层 `groups=`** *（强，但仅限视图）* —— 列表列、看板卡片区段、表单页签上带 `groups=` 的元素，会在服务端从视图 arch 中移除后再发送给客户端。无权限用户根本不会收到这些元素。

### 表单层视觉隐藏的限制

联系人**表单**上的电话和邮箱可见性通过 `invisible=` 表达式控制，而非 `groups=`。这是经过权衡的取舍：原生 Odoo 的 `phone` 和 `email` 字段没有模型层 `groups`，给它们加上模型层 `groups` 会破坏所有依赖这些字段的其他模块（CRM、销售、会计、WhatsApp 等）的兼容性。

实际后果是：

> 技术上熟练的用户（浏览器 DevTools、自定义 RPC 客户端）仍能读取到表单层 `invisible=` 隐藏的电话和邮箱值。**列表和看板**视图使用字段层 `groups=`，会在服务端剥离数据，所以列层级的隐藏是数据严密的；**表单层**的隐藏只是 UI 层面的。

对大多数 B2B 团队这是可接受的——这条规则把日常 UI 中销售员之间互相浏览客户的便利性挡掉了，这才是真实的威胁模型。下一节会解释为什么我们没有更进一步用记录规则把表单隐藏做成数据严密的。

### 为什么不用记录规则？

#### 问题出在哪

Odoo 中实现"销售员只看自己客户"的主流方式是给 `res.partner` 加一条 `ir.rule`，比如 `[('user_id', '=', user.id)]`。Apps Store 上几乎所有付费模块和论坛上大部分回答都走这条路。

我们先试了这条路，然后放弃了。原因如下。

`res.partner` 是 Odoo 的中央引用枢纽。用户、员工、公司、银行账户、关注者、会计分录、消息作者、销售订单的发货和发票地址、产品上的供应商信息——所有这些都是 partner，或者通过 partner 关联。一旦你用记录规则收紧 `res.partner` 的读取权限，所有*隐式*加载关联 partner 的地方都可能抛出 `AccessError`。实际表现：

- 销售员打不开自己的报价单，因为 SO 关联的 `partner_invoice_id` / `partner_shipping_id` 销售员读不到
- 登录失败，因为 `res.users` 反向引用用户自己的 partner 记录
- 聊天、关注者、活动、被分配字段抛 `Implicitly accessed through 'Users' (res.users)` 错误
- 多公司 partner（`res.company` 的伴随 partner）变得不可见，破坏仓库、日记账的路由

#### 大家都尝试过的变通方案

针对这些错误，现有的解决方案都不优雅：编辑内置的 `res.partner.rule.private.employee` 规则，把内部用户拆成两个并行群组（一个有规则一个没有），给记录规则上每出现一种新边界情况就再加一个 `OR` 子句，把 `enforce` 降级为只读以避免创建失败……每一种变通都在漏。Odoo 后续升级、新装的模块、之前少见的 partner 引用，随时可能让访问错误重新冒出来。Odoo 自己在 v17 中移除了原本的 `res.partner.rule.private` 规则，原因正是这种设计造成的回归太多。

#### 我们的不同立场

Contact Guard 选择了不同的立场。**联系人记录对所有内部用户保持完全可见。** 销售流程不会被破坏。电话、邮箱、销售员才是你真正想保护的东西，而这些通过字段层 `groups=`、ORM 写入钩子、视图层 `groups=` 来管控（参见上面的强制执行层级）。这个立场的代价是表单层的电话/邮箱 `invisible=` 只是 UI 层面而非数据严密的，但我们认为这对大多数 B2B 团队是正确的取舍：没人会打开 DevTools 去爬同事的客户，但每天都有人因为搜索框联想里弹出同事的客户而误打过去。

如果你的安全模型确实需要 `res.partner` 的记录级隔离（受监管的数据、需要审计的环境），Contact Guard 不是合适的工具——直接用记录规则，承担维护负担。除此之外的场景，本模块就是要给你 80% 的解决方案，不带那 20% 的破坏。

## 兼容性

- Odoo 19.0（Community 和 Enterprise）
- 多公司感知：每项校验和配置都按公司隔离
- 不引入数据库层级约束；强制执行均在 ORM 层、字段层或视图层
- 仅依赖 Odoo 标准模块：`contacts`、`mail`、`sales_team`、`sale`、`purchase`

## 安装

### 通过 Odoo Apps Store 安装

在 Apps 菜单中搜索 **Contact Guard** 安装即可。这是推荐方式。

### 通过源码安装

```bash
# 克隆 SuiteState community 模块单仓（19.0 分支）
git clone -b 19.0 https://github.com/SuiteState/community.git

# 复制或软链接 suite_contact_guard 到 Odoo 的 addons 路径
cp -r community/suite_contact_guard /path/to/odoo/addons/

# 在 Odoo 中：Apps -> 更新应用列表 -> 搜索 "Contact Guard" -> 安装
```

依赖（标准 Odoo 模块）：

- `contacts`
- `mail`
- `sales_team`
- `sale`
- `purchase`
