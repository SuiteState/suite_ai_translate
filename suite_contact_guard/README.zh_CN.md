# Contact Guard

Odoo 19 联系人模块的电话号码唯一性、字段权限管控与客户活跃度状态追踪。

## 概述

Contact Guard 在 `res.partner` 上添加三层互相独立的管控机制：

1. ORM 层级的电话号码唯一性校验，按公司隔离
2. 基于角色的可见性与编辑权限管控（电话、邮箱、销售员字段）
3. 每日定时任务计算的客户活跃度状态（活跃 / 警告 / 沉睡 / 流失），基于已确认的销售订单

第四个支持性特性处理 `company_id` 默认值，让单公司部署获得与多公司部署相同的按公司隔离行为。

四项功能均按公司配置。所有功能均不引入数据库层级的约束；强制执行发生在 ORM 层、字段层或视图层（视具体规则而定）。详见[权限 → 强制执行层级](#强制执行层级)。

## 设计原则

本模块遵循以下设计原则。这些原则用于解释后续章节中的技术决策。

- **联系人记录全局可读。** 所有内部用户均可读取 `res.partner` 记录。本模块不通过 `ir.rule` 限制联系人记录的读取权限。
- **跨部门工作流不受影响。** 会计、仓库、采购、人力资源、CRM 等模块通过其常规引用（发票地址、送货地址、供应商记录、员工伴随联系人、商机联系人）继续正常访问联系人数据。
- **保护施加于字段层级。** 受保护的字段是电话、邮箱、销售员。可见性与编辑规则按字段单独施加，通过字段层 `groups=`、ORM 写入钩子、视图层 `groups=` 三种机制实现。
- **按公司隔离。** 每项校验、阈值、配置都按单一公司隔离。多公司部署中，每家公司有独立的配置。

## 功能特性

### 电话号码唯一性

电话号码会被规范化为纯数字（去除 `+`、前导 `00`、空格、连字符、括号），存入索引字段供查重使用。

当联系人被创建或其电话字段被更新为已存在于同公司内的值时，操作会被拒绝，错误信息会指明冲突的号码。

以下场景会跳过唯一性校验：

- `env.su`（系统自动化、迁移、商机转联系人同步）
- 通过标准导入向导进行的批量导入（`import_file` context）
- Public 用户（匿名自助注册访问者）
- Portal 用户（外部客户管理自身数据）
- 显式传入的 `suite_skip_phone_check` context flag

最小位数可按公司配置，默认 8 位。

### 联系人字段权限

电话、邮箱、销售员字段的可见性按角色管控。一旦联系人被分配了销售员，其电话和邮箱字段就只对该销售员和管理员可访问。未分配销售员的联系人对所有内部用户保持可访问。

销售员字段在创建时可以自由设置（支持 CRM 商机转联系人时携带销售员信息）。创建后，仅 Contact Guard 管理员可重新分配销售员。

电话编辑保护在 ORM 层强制执行。该保护适用于所有代码路径，包括 Web UI、RPC 调用、脚本、自动化操作。

完整的角色权限矩阵详见下方的[权限](#权限)一节。

### 客户活跃度状态

按公司启用后，每日定时任务会根据每位客户最后一笔已确认销售订单的日期计算其活跃度状态：

| 状态 | 条件 |
|---|---|
| 活跃（Active） | 距上次订单天数 < 警告阈值 |
| 警告（Warning） | 警告阈值 <= 天数 < 沉睡阈值 |
| 沉睡（Sleeping） | 沉睡阈值 <= 天数 < 流失阈值 |
| 流失（Dormant） | 天数 >= 流失阈值 |
| 空白 | 没有已确认的销售订单 |

空白状态通过独立的 `suite_has_orders` 布尔字段追踪，让"刚下单 0 天"和"从未下单"在视图中保持可区分，同时让天数字段保持为普通整数，列表视图中能正确排序。

阈值可按公司配置。状态显示在联系人表单（活跃度状态标签页）和列表视图中（带颜色装饰：黄色=警告、红色=沉睡、灰色=流失），可用于筛选和分组。

定时任务以 1000 条为一批处理客户，关闭聊天通知。

### 多公司 `company_id` 默认值

在原生 Odoo 中，单公司用户的联系人表单上不显示 `company_id` 字段，且 Odoo 默认将其设为空（这意味着该联系人对所有公司全局可见）。

本模块在创建联系人时自动注入用户当前所属公司作为默认 `company_id`，让单公司部署获得与多公司部署手动配置相同的按公司隔离效果。

当联系人是因为新建 `res.company` 而被自动创建为伴随主联系人时，这项注入会被禁用。在那种情况下，Odoo 原生流程会分配正确的 company id，否则注入会用当前激活公司的 id 覆盖它，破坏新公司的下游路由。

## 配置

进入：

> **联系人 -> 配置 -> Contact Guard**

（默认仅系统管理员可见。）

保护功能默认开启，不需要配置记录就能工作。电话号码唯一性、字段权限、销售员锁定在模块安装的瞬间就开始生效。配置仅用于调整参数或启用活跃度追踪。

| 功能 | 默认状态 | 可配置参数 |
|---|---|---|
| 电话号码唯一性 | 开启 | 最小位数（默认 8） |
| 字段权限 | 开启 | 无；行为由角色固定决定 |
| 客户活跃度状态 | 关闭 | 启用定时任务，警告 / 沉睡 / 流失阈值 |

按公司的设置项：

| 设置项 | 默认值 | 说明 |
|---|---|---|
| Phone Min Length | 8 | 规范化后的最小位数 |
| Track Customer Activity | 关闭 | 启用本公司的每日活跃度定时任务 |
| Warning after (days) | 30 | 进入警告状态的天数 |
| Sleeping after (days) | 60 | 进入沉睡状态的天数 |
| Dormant after (days) | 90 | 进入流失状态的天数 |

阈值必须满足 `0 < 警告 < 沉睡 < 流失`。

## 权限

### 角色

| 角色 | 群组 | 来源 |
|---|---|---|
| Contact Guard 管理员 | `suite_contact_guard.group_contact_guard_admin` | 本模块新增 |
| 销售经理 | `sales_team.group_sale_manager` | 原生；蕴含 Contact Guard 管理员 |
| 联系人管理员 | `base.group_partner_manager` | 原生 |
| 采购经理 | `purchase.group_purchase_manager` | 原生 |
| 采购用户 | `purchase.group_purchase_user` | 原生 |
| 该联系人的销售员 | `partner.user_id == self.env.user` | 按记录的关系 |

销售经理群组蕴含 Contact Guard 管理员（单向蕴含）。单独授予 Contact Guard 管理员不会同时授予销售经理权限。这一设计允许将 Contact Guard 管理员分配给销售体系外的角色，而不会同时授予销售经理权限。

### 权限矩阵

下表中，*销售员*指被分配为该联系人销售员的用户。*SM/CGA* 指销售经理或 Contact Guard 管理员（在 Contact Guard 操作上等价）。

| 操作 | 内部用户 | 该联系人的销售员 | 采购经理 | 联系人管理员 | SM / CGA |
|---|---|---|---|---|---|
| 在列表和看板中看到电话/邮箱列 | ❌ | ❌ | ❌ | ✅ | ✅ |
| 在表单上看到电话/邮箱*（联系人无销售员）* | ✅ | – | ✅ | ✅ | ✅ |
| 在表单上看到电话/邮箱*（联系人有销售员）* | ❌ ¹ | ✅ | ❌ ¹ | ✅ | ✅ |
| 首次填写电话（原本为空） | ✅ | ✅ | ✅ | ✅ | ✅ |
| 编辑已有电话*（联系人无销售员）* | ❌ | ❌ | ✅ | ✅ | ✅ |
| 编辑已有电话*（联系人有销售员）* | ❌ | ❌ | ❌ | ✅ | ✅ |
| **创建**联系人时设置销售员 | ✅ | ✅ | ✅ | ✅ | ✅ |
| 修改已有联系人的销售员 | ❌ | ❌ | ❌ | ❌ | ✅ |
| 看到活跃度状态（表单标签页 + 列表列） | ❌ | ❌ | ❌ | ✅ | ✅ |
| 打开 Contact Guard 配置菜单 | ❌ | ❌ | ❌ | ❌ | 仅系统管理员 |
| 写入 Contact Guard 配置 | ❌ | ❌ | ❌ | ❌ | 仅系统管理员 |

> ¹ 联系人有销售员时，"内部用户"和"采购经理"在表单上看不到电话/邮箱*值*。该字段通过 `invisible=` 隐藏，并不是在服务端剥离。具备技术访问能力的用户（DevTools、自定义 RPC 客户端）可以读取该值。列表和看板的隐藏*是*在服务端剥离的。详见[表单层可见性的范围与限制](#表单层可见性的范围与限制)。

### 强制执行层级

本模块在三个层级强制执行规则：

1. **ORM 层** —— 电话编辑保护和销售员变更保护在 `res.partner.write` 中强制执行。适用于 Web UI、RPC、脚本、自动化操作。仅在 `env.su`、`import_file` context、`suite_skip_phone_check` context、public/portal 用户场景下绕过。
2. **字段层 `groups=`** —— 活跃度状态字段（`suite_last_order_date`、`suite_days_since_order`、`suite_activity_status`、`suite_has_orders`）在模型上携带 `groups=`。无权限用户在 `search_read`、`read`、导出、API 调用中均无法获取这些字段值，ORM 会在服务端剥离。
3. **视图层 `groups=`** —— 列表列、看板卡片区段、表单页签上带 `groups=` 的元素，会在服务端从视图 arch 中移除后再发送给客户端。

## 实现说明

### 表单层可见性的范围与限制

联系人**表单**上的电话和邮箱可见性通过 `invisible=` 表达式控制，而非字段层 `groups=`。这是经过权衡的设计决策。原生 Odoo 的 `phone` 和 `email` 字段没有模型层 `groups`，给它们添加模型层 `groups` 会破坏所有依赖这些字段的其他模块（CRM、销售、会计、WhatsApp 等）的兼容性。

实际后果：

- **列表和看板**视图在列上使用字段层 `groups=`，在服务端剥离数据。列层级的隐藏是数据严密的。
- **表单**视图使用 `invisible=`。字段值会被发送到客户端，由 UI 隐藏。具备技术访问能力的用户（浏览器 DevTools、自定义 RPC 客户端）可以读取这些值。

如果部署需要表单层数据严密的保护，应当使用 record rules 而非本模块。这一权衡详见下一节。

### 对比：字段层方案 vs `res.partner` 上的 record rules

Odoo 中实现联系人隐私保护有两种技术方案：通过 `res.partner` 上的 `ir.rule` 限制记录可见性，或通过字段层 `groups=` 和 ORM 钩子限制字段可见性。本模块采用后者。

| 方面 | `res.partner` 上的 record rule | 字段层 + ORM 钩子（本模块） |
|---|---|---|
| 列表和看板中隐藏电话/邮箱列 | 是 | 是（通过字段 `groups=` 在服务端剥离） |
| 表单上隐藏电话/邮箱值 | 数据严密（记录不加载） | UI 层级，通过 `invisible=` |
| 销售员打开自己的报价单（引用 partner） | 需要在 `partner_invoice_id`、`partner_shipping_id` 上添加 `OR` 子句 | 无需修改即可工作 |
| 用户登录（`res.users` 自引用） | 需要为自引用添加显式 carve-out | 无需修改即可工作 |
| 多公司 partner（`res.company` 的伴随 partner） | 需要添加显式 `OR` 子句 | 无需修改即可工作 |
| 在 partner 关联记录上的聊天、关注者、活动分配 | 需要添加 carve-out 以避免 `Implicitly accessed through 'Users'` 错误 | 无需修改即可工作 |
| 跨部门 partner 访问（会计、仓库、采购、HR） | 限制为记录所有者；每个跨部门工作流需要按部门添加 carve-out | 无限制；partner 记录保持可读，电话/邮箱/销售员按角色管控 |
| Odoo 升级或安装新模块后的维护 | 当关联模型新增 partner 引用时，可能需要重新验证 carve-out | 稳定；依赖在各版本中保持稳定的 Odoo API（`groups=`、`invisible=`、写入钩子） |

### 跨部门 partner 访问

在典型的 Odoo 部署中，partner 记录被多个模块引用：

- **会计** —— 发票地址、付款 partner、银行账户持有人
- **仓库 / 库存** —— 送货地址、入库单上的供应商记录
- **采购** —— 采购订单和账单上的供应商记录
- **HR** —— 员工记录和相关联系人
- **CRM** —— 商机和销售机会的 partner

本模块采用的字段层方案保持跨部门 partner 访问不受影响。会计用户继续访问发票地址。仓库用户继续访问送货地址。采购用户继续访问供应商记录。受保护字段（电话、邮箱、销售员）按角色管控，独立于 partner 记录访问。

## 兼容性

- Odoo 19.0（Community 和 Enterprise）
- 多公司感知：每项校验和配置都按公司隔离
- 不引入数据库层级约束；强制执行均在 ORM 层、字段层或视图层

## 安装

### 通过 Odoo Apps Store 安装

在 Apps 菜单中搜索 **Contact Guard** 安装即可。

### 通过源码安装

```bash
# 克隆 SuiteState community 模块单仓（19.0 分支）
git clone -b 19.0 https://github.com/SuiteState/community.git

# 复制或软链接 suite_contact_guard 到 Odoo 的 addons 路径
cp -r community/suite_contact_guard /path/to/odoo/addons/

# 在 Odoo 中：Apps -> 更新应用列表 -> 搜索 "Contact Guard" -> 安装
```
